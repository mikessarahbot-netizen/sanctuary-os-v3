# DB Plan

## Purpose
Provide production persistence for Sanctuary OS while keeping tenant scope, auditability, offline-compatible domain boundaries, and adapter isolation explicit.

## First production target
Planning server persistence is the first DB implementation target because the Planning API contract layer is complete enough to exercise repository boundaries end to end.

Initial implementation order:
1. Planning command persistence repository.
2. Planning query persistence repository.
3. Planning rehearsal tracking persistence repository.
4. Planning CCLI usage persistence repository.
5. Planning readiness persistence storage and lookup.

## Adapter strategy
Use `packages/db` as the persistence contract and adapter package. The first production adapter should be PostgreSQL-backed for API server usage and should implement the existing repository interfaces from `packages/db/src/planning-repository-contracts.ts`.

Do not put vendor SDK calls, GraphQL resolver logic, Auth0 claims handling, or API service behavior in `packages/db`. The DB package owns database connection boundaries, migrations, schema mapping, transaction handles, tenant-scoped repository adapters, and persistence-level validation.

## Tooling decision gate
ADR 0002 deferred ORM/query-builder/migration-tool choice. Before writing production adapter code, create a short ADR that chooses one of:
- SQL-first adapter with a small PostgreSQL client and explicit SQL migrations.
- Typed query-builder adapter.
- ORM-backed adapter.

Decision criteria:
- Strong tenant-scope guarantees are easy to review.
- Migrations are deterministic and reviewable.
- Transaction handles can be passed through repository operations.
- Tests can run locally without secrets.
- SQLite/local compatibility is not made harder for future Play and Charts work.

Until that ADR exists, implementation slices may add plan docs, test fixtures, interface refinements, or adapter acceptance tests only.

## PostgreSQL server persistence
The PostgreSQL adapter must support:
- Tenant-scoped reads and writes for every Planning persistence operation.
- Stable IDs generated by the adapter or supplied by a domain-safe ID boundary.
- UTC timestamps.
- Audit records for every successful mutation.
- Confirmation-intent audit metadata for publish/cancel flows.
- Atomic updates for service status, item reorder, assignment status, CCLI usage writes, rehearsal visibility writes, and rehearsal acknowledgement writes.
- Read indexes for service lists, service item ordering, assignment lookup, readiness lookup, CCLI usage by service/status, rehearsal visibility by service/item, and acknowledgements by service/item/person/assignment/asset.

The PostgreSQL adapter must validate incoming persistence operations with the existing Zod operation schemas before touching storage or must normalize all persistence-layer validation failures into stable adapter errors.

## SQLite and local compatibility
SQLite is a future local/offline target for Play, Charts, and potentially desktop Planning cache. The server PostgreSQL schema should avoid assumptions that make future SQLite projections awkward:
- Keep tenant IDs explicit on persisted records.
- Keep service, item, assignment, song, asset, CCLI, and rehearsal IDs opaque strings at the repository boundary.
- Avoid PostgreSQL-only domain encoding in public repository contracts.
- Keep ordering, timestamps, booleans, and enums representable in SQLite.
- Do not couple local sync queues to the first PostgreSQL adapter slice.

SQLite implementation is out of scope for the first DB adapter. Future local/offline slices must include sync-queue and stale-data tests as required by `02-standards/engineering-rules.md`.

## Migration boundaries
The DB package should own migration registry contracts and production migration files once tooling is chosen.

Migration expectations:
- Every migration has a stable ID, checksum, and applied/rollback state.
- Migrations are reviewed as code and do not require secrets in the repo.
- Migration tests verify schema shape and rollback behavior where the chosen tool supports it.
- Initial Planning migrations should create only the tables needed by implemented Planning persistence interfaces and audit metadata.
- Destructive migrations require an explicit ADR and a rollout plan.

Initial Planning table groups:
- `planning_services`
- `planning_service_items`
- `planning_assignments`
- `planning_service_templates`
- `planning_song_library_items` or an adapter view over the song catalog boundary
- `planning_readiness_results`
- `planning_ccli_usage_logs`
- `planning_rehearsal_asset_visibility`
- `planning_rehearsal_acknowledgements`
- `planning_audit_log`

## Transaction behavior
The adapter must honor `options.transaction` from repository operations. If a transaction handle is supplied, the adapter must use it and must not create an independent transaction.

Operations requiring atomicity:
- Publish/cancel service updates plus confirmation audit metadata.
- Service item reorder.
- Assignment status update plus audit metadata.
- Readiness save.
- CCLI usage record.
- Rehearsal asset visibility set.
- Rehearsal acknowledgement record.

Events remain API-owned and are published through `publishAfterCommit`; DB adapters must not publish WebSocket events directly.

## Tenant scoping
Every table that stores tenant-owned Planning data must include `tenantId` or its selected database column equivalent. Every repository method must filter by `options.context.tenantId` before reading or writing.

Rules:
- Do not infer tenant scope from IDs.
- Do not return cross-tenant records.
- Reject writes when related service, item, assignment, role/person reference, song reference, asset visibility, or acknowledgement records do not belong to the requested tenant.
- Returned records must include the same tenant ID supplied in operation context.

## Audit metadata
Every successful mutation must persist:
- Tenant ID.
- Actor ID when supplied.
- Request ID.
- Repository method or operation name.
- Mutation intent.
- Timestamp.
- Target aggregate ID where available.
- Confirmation reason for publish/cancel flows.

Audit records must not store secrets, raw prompt payloads, volunteer contact details, raw media payloads, or vendor credentials.

## Error model
The adapter should normalize persistence failures into stable error categories before surfacing them to services:
- Validation failure.
- Tenant-scope mismatch.
- Not found.
- Conflict or stale write.
- Transaction failure.
- Connection unavailable.

Detailed database errors may be logged only after redacting secrets and raw PII. User-facing API errors remain an API concern.

## Test strategy
Add tests before each production adapter slice:
- Contract tests that run the adapter through existing Planning service tests where practical.
- Repository-level tests for tenant isolation, cross-tenant misses, mutation audit metadata, transaction handle usage, and malformed operation rejection.
- Migration tests for schema creation and required indexes.
- Failure tests for unavailable database and transaction rollback.

Local tests must not require checked-in secrets. If a live PostgreSQL instance is needed, tests must either use a local ephemeral database setup documented in the package README or skip with a clear message unless required environment variables are present.

## Rollout order
1. Write ADR for database adapter and migration tooling.
2. Add migration framework and Planning schema migration tests.
3. Implement Planning command repository adapter for service, item, assignment, and template operations.
4. Implement Planning query repository adapter.
5. Implement rehearsal tracking repository adapter.
6. Implement CCLI usage repository adapter.
7. Implement readiness persistence adapter.
8. Wire API composition to select in-memory/test or production adapters by environment.
9. Run a DB release-check before using production persistence from app surfaces.

## Out of scope
- Concrete adapter code in this planning slice.
- ORM/query-builder installation.
- Connection strings or secrets.
- Production migrations before the tooling ADR.
- UI, GraphQL resolver, service, vendor integration, queue worker, and notification changes.
