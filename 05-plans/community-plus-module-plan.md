# Community+ Module Plan

## Scope v1
Tenant-scoped **member directory** (people the church tracks: name, segment/status, custom-field values, contact-channel references) · **households** that group members · **groups/teams** as named member collections (small group, serving team, ministry) with memberships · **attendance** records for a service/gathering occasion (per-member present/absent/excused, plus an anonymous headcount) · **communications** as a *draft → reviewed → confirmed → queued-for-send* outbound message model to a recipient set (the **send itself is an integration concern**, gated behind explicit human confirmation; Community+ owns the message record and its send-intent lifecycle, never the carrier) · **engagement** as derived, **non-PII** rollups (attendance streaks, serving frequency, comms-response counts) used for AI-safe projections · references to Planning services and `teamProfile`/`peopleProfile` members by **opaque IDs**.

## Out of scope v1
General-purpose CRM (pipelines, deals, arbitrary record types) — a stated non-goal · giving/donations/pledges and any individual financial data (non-goal; **never modeled here**) · child check-in / child-sensitive records as a feature (the *category* is named only to be **excluded from AI** and flagged as a future, separately-governed module) · pastoral-care / counseling / prayer-request **note content** as a stored AI-reachable field (if a prayer-request *object* lands later it is private-by-construction; see Privacy) · the actual delivery transport (Twilio SMS, email, push) — Community+ produces a confirmed send-intent and hands off to an integration adapter; it does not own carrier SDKs, credentials, delivery receipts, or phone/email *values* in domain records · two-way inbound message threads / a full inbox · marketing automation, drip campaigns, A/B sends · self-service member portal / public sign-up forms · realtime multi-admin co-editing · AI auto-sending of any communication (forbidden — AI may draft, never send) · contact-value storage (raw phone numbers, home addresses, emails) inside Community+ domain records (held only as opaque channel references resolved by a separate, access-controlled contact-vault boundary; see Privacy).

## Domain objects
Member · Household · CommunityGroup · GroupMembership · AttendanceRecord · CommunicationMessage · CommunicationRecipient · EngagementSummary

## Domain boundaries
| Object | Owns | Does not own |
|---|---|---|
| Member | Tenant-scoped member identity, display name, status/segment, household ref, custom-field **values**, opaque contact-channel refs, consent flags | Raw contact values (phone/email/address), giving data, prayer/counseling notes, auth identity, the custom-field *schema* (that is `peopleProfile`) |
| Household | Tenant-scoped household identity, label, member refs, primary-contact member ref | Member personal data itself, addresses, billing |
| CommunityGroup | Tenant-scoped group/team identity, kind (small-group/serving-team/ministry/class/other), label, optional leader member ref | Member PII, Planning assignments, scheduling policy (that is `teamProfile`) |
| GroupMembership | Membership edge (member ↔ group), role-in-group, joinedAt/active | Member identity, group identity, attendance |
| AttendanceRecord | Tenant-scoped attendance for one (occasion, member?) — status present/absent/excused, or an anonymous headcount row | The service definition (Planning owns it), comms, giving |
| CommunicationMessage | Outbound message **record**: channel kind, subject/body template, audience descriptor, lifecycle status (draft/reviewed/confirmed/queued/sent/failed/canceled), confirmation metadata, audit | The carrier/transport, credentials, delivery receipts, raw recipient contact values, message *content addressed to AI* |
| CommunicationRecipient | Resolved recipient edge for a message: member ref + opaque channel ref + per-recipient send status | The contact value itself, the carrier, the member record |
| EngagementSummary | Derived **non-PII** rollups per member or segment (attendance streak, serving count, last-present occasion ref, comms-response count) | Source PII, the raw events it aggregates, any free-text, giving |

## API and storage surfaces
- GraphQL/HTTP is the cross-device persistence boundary for members, households, groups, memberships, attendance, communications, and engagement summaries (mirrors Charts/Play/Presenter). Community+ is **web-admin primary** (system map: "Web app … community workflows").
- Community+ is **primarily online** (admin workflows on the web app), not offline-first. The system-map failure strategy lists only Play + Presenter as continuing locally; the offline-first engineering rule names only Play and Charts. Community+ therefore does **not** ship a local sync queue in v1 — see *Offline posture* for the justification and the one narrow exception considered.
- **Contact values are not stored in Community+.** A member references contact channels by opaque IDs (`contactChannelRef`) resolved by a separate access-controlled contact-vault boundary (out of scope here, same posture as Play media refs / Charts song refs). Domain records hold references and consent flags, never phone/email/address strings.
- Custom-field **schema** lives in `peopleProfile` (ChurchContext); Community+ stores only validated **values** keyed by opaque field IDs and never invents fields.
- Communications produce a **confirmed send-intent**; an integration adapter (Twilio/email, isolated per the API-shape rule) realizes the actual send and reports per-recipient status back. Community+ never holds carrier credentials and never auto-sends.
- Service/gathering and member references are opaque IDs from Planning / `peopleProfile` / `teamProfile`; Community+ reads but does not mutate them.

## Domain objects as records (Zod-shaped, tenant-scoped; PII-segregated; nothing AI-reachable carries PII)
Every record is `.strict()`, branded-ID typed at boundaries, carries `tenantId`, and stores only **references and minimal values** — never raw contact values, credentials, giving data, or care/counseling note text. Field lists name the durable shape the API domain (`apps/api/src/domain/community/contracts.ts`) and persistence records (`packages/db/src/community-repository-contracts.ts`) must agree on. **PII fields are explicitly marked `[PII]`**; every `[PII]` field is excluded from any value that reaches an AI prompt (see Privacy & safety and AI assist).

- **Member** — `tenantId`, `memberId`, `householdRef?`, `displayName` **[PII]**, `status` (enum: active/inactive/visitor/archived), `segmentRefs: string[]` (opaque segment IDs from `peopleProfile`), `customFieldValues: CustomFieldValue[]` (each: `fieldRef` opaque ID, `value` string — **treated [PII] in aggregate**, never sent to AI), `contactChannelRefs: ContactChannelRef[]` (each: `channelRef` opaque vault ID, `kind` ∈ {sms, email, push, other}, `consentStatus` ∈ {granted, denied, unknown}) **[PII-bearing reference]**, `createdAt`, `updatedAt`.
  - Invariants: `displayName` non-empty; `contactChannelRefs` unique by `channelRef`; `customFieldValues` unique by `fieldRef`; **no raw contact value present** (only `channelRef`); `status` from the fixed enum.
- **Household** — `tenantId`, `householdRef`, `label` **[PII — often a family name]**, `memberRefs: string[]`, `primaryContactMemberRef?`, `createdAt`, `updatedAt`.
  - Invariants: `memberRefs` unique and non-empty when persisted; `primaryContactMemberRef`, if set, appears in `memberRefs`.
- **CommunityGroup** — `tenantId`, `groupId`, `kind` (enum: small-group/serving-team/ministry/class/other), `label`, `leaderMemberRef?`, `archived` bool, `createdAt`, `updatedAt`.
  - Invariants: `label` non-empty; `kind` from the fixed enum; group label is **non-PII** (a ministry name), so groups may appear in AI projections while their *members* may not.
- **GroupMembership** — `tenantId`, `membershipId`, `groupId`, `memberRef`, `roleInGroup` (enum: leader/co-leader/member/guest), `active` bool, `joinedAt`, `updatedAt`.
  - Invariants: one active membership per `(groupId, memberRef)`; `roleInGroup` from the fixed enum.
- **AttendanceRecord** — `tenantId`, `attendanceId`, `occasionRef` (opaque service/gathering ID), `memberRef?` (omitted ⇒ this is an anonymous headcount row), `status` (enum: present/absent/excused), `headcount?` (positive int — present **iff** `memberRef` absent), `recordedByRef`, `recordedAt`, `updatedAt`.
  - Invariants (superRefine): exactly one of {`memberRef` + `status`} **or** {`headcount` anonymous row} — `memberRef` present ⇒ `headcount` absent and `status` present; `memberRef` absent ⇒ `headcount` present (`> 0`). One member row per `(occasionRef, memberRef)`. `headcount > 0`.
- **CommunicationMessage** — `tenantId`, `messageId`, `channel` (enum: sms/email/push), `subject?` (email only), `bodyTemplate` (text with `{{placeholder}}` tokens — **no resolved PII baked in**; recipients are merged at send time by the integration adapter, not stored expanded), `audience: AudienceDescriptor` (one of: `{ kind: "group", groupId }` | `{ kind: "segment", segmentRef }` | `{ kind: "explicit", memberRefs[] }`), `status` (enum: draft/reviewed/confirmed/queued/sent/failed/canceled), `confirmation?: { confirmed: true, confirmedByRef, reason, confirmedAt }` (**required to leave `reviewed`**), `origin` (enum: human/ai-drafted — ai-drafted still requires human confirmation to advance), `createdByRef`, `createdAt`, `updatedAt`.
  - Invariants (superRefine): `channel = email` ⇒ `subject` allowed; `channel ≠ email` ⇒ `subject` absent. Status may advance to `confirmed`/`queued`/`sent` **only if** `confirmation.confirmed = true`. `bodyTemplate` stores **template tokens, never expanded recipient PII**. `origin = "ai-drafted"` never auto-advances past `draft` without human `confirmation`. Allowed transition map enforced (draft→reviewed→confirmed→queued→sent, with →canceled / →failed terminal branches); illegal jumps rejected.
- **CommunicationRecipient** — `tenantId`, `recipientId`, `messageId`, `memberRef`, `channelRef` (opaque vault ID) **[PII-bearing reference]**, `sendStatus` (enum: pending/sent/delivered/failed/suppressed), `failureReason?` (redacted/safe — **never raw carrier payload or contact value**), `updatedAt`.
  - Invariants: one recipient row per `(messageId, memberRef)`; `failureReason` present only when `sendStatus = failed`; holds only `channelRef`, never the contact value; `suppressed` when consent is denied/unknown for the channel.
- **EngagementSummary** — `tenantId`, `summaryId`, `scope` (one of: `{ kind: "member", memberRef }` | `{ kind: "segment", segmentRef }`), `attendanceStreak` (nonneg int), `lastPresentOccasionRef?`, `servingCount` (nonneg int), `commsResponseCount` (nonneg int), `windowStart`, `windowEnd`, `computedAt`.
  - Invariants: all counts `>= 0`; `windowEnd >= windowStart`; **contains no names, no contact data, no free-text** — only refs and counts, so it is the one Community+ record class that is AI-projectable at the member-scope level (subject to `aiPolicyProfile`).

## Pure rendering/transform rules
The Community+ analog of Charts `chordpro.ts` / Play `sequence.ts` lives in `apps/api/src/domain/community/` as pure, deterministic, Zod-validated, no-I/O functions (no `Date.now`, no randomness; clocks injected at the service edge).

- **Audience resolution** (`audience.ts`) is a pure function: `(AudienceDescriptor, Member[], GroupMembership[]) → ResolvedAudience` — expands a group/segment/explicit descriptor to a **deduplicated, consent-filtered set of `{ memberRef, channelRef }`**, dropping members with `consentStatus ≠ granted` for the message channel and flagging them as `excludedForConsent` (passed through flagged, never silently dropped — same posture as flagged-unresolved in Play/invalid-chord-passthrough in Charts). Returns **refs only**, never contact values or names.
- **Engagement rollup** (`engagement.ts`) is a pure function: `(attendanceRows, membershipRows, recipientRows, window) → EngagementSummary[]` — computes streaks/counts deterministically over an explicit window; emits only refs + counts; provably PII-free by construction (its output type cannot hold names/contact data).
- **Message lifecycle transition** (`message-lifecycle.ts`) is a pure function: `(CommunicationMessage, transition, confirmation?) → CommunicationMessage | LifecycleError` — validates the allowed-transition map and the *confirmation-required-to-send* invariant as a pure state machine (discriminated-union friendly, per the engineering rule preferring discriminated unions for stateful workflows). No send, no I/O — it only decides whether the move is legal.
- **Attendance tally** (`attendance.ts`) is a pure function: `(AttendanceRecord[]) → AttendanceTally` — sums present/absent/excused + anonymous headcount per occasion into a display/rollup projection; never mutates source rows.
- Determinism + scope: all take already-tenant-scoped records and are tenant-agnostic at the function level; deterministic; injected clock only at the service edge.

## Persistence model (SQLite-compatible: TEXT/INTEGER/REAL)
New file `packages/db/src/community-migrations.ts` exporting `CommunityInitialSchemaMigration` (via `defineSqlMigrationArtifact`), aggregated as `CommunitySqlMigrations`, mirroring `charts-migrations.ts` / `play-migrations.ts`. All tables tenant-scoped, composite PKs lead with `tenant_id`, `CHECK` constraints encode the Zod invariants, booleans as `INTEGER IN (0,1)`, counts as `INTEGER`, timestamps/enums/JSON as `TEXT`. **No column ever stores a raw contact value, credential, giving figure, or care-note text** — references and minimal values only.

Initial schema tables (`schema_version` literal `community.v1`):
- `members` — PK `(tenant_id, member_id)`; columns: `household_ref TEXT`, `display_name TEXT NOT NULL`, `status TEXT NOT NULL`, `segment_refs_json TEXT NOT NULL DEFAULT '[]'`, `custom_field_values_json TEXT NOT NULL DEFAULT '[]'`, `contact_channel_refs_json TEXT NOT NULL DEFAULT '[]'`, `schema_version TEXT NOT NULL`, `created_at TEXT NOT NULL`, `updated_at TEXT NOT NULL`; `CHECK (display_name <> '')`, `CHECK (status IN ('active','inactive','visitor','archived'))`, `CHECK (schema_version = 'community.v1')`. Index `members_tenant_status_idx (tenant_id, status)`, `members_tenant_household_idx (tenant_id, household_ref)`.
- `households` — PK `(tenant_id, household_ref)`; `label TEXT NOT NULL`, `member_refs_json TEXT NOT NULL DEFAULT '[]'`, `primary_contact_member_ref TEXT`, `created_at`, `updated_at`; `CHECK (label <> '')`. Index `households_tenant_idx (tenant_id)`.
- `community_groups` — PK `(tenant_id, group_id)`; `kind TEXT NOT NULL`, `label TEXT NOT NULL`, `leader_member_ref TEXT`, `archived INTEGER NOT NULL DEFAULT 0`, `created_at`, `updated_at`; `CHECK (kind IN ('small-group','serving-team','ministry','class','other'))`, `CHECK (archived IN (0,1))`, `CHECK (label <> '')`. Index `community_groups_tenant_kind_idx (tenant_id, kind)`.
- `group_memberships` — PK `(tenant_id, membership_id)`; `group_id TEXT NOT NULL`, `member_ref TEXT NOT NULL`, `role_in_group TEXT NOT NULL`, `active INTEGER NOT NULL DEFAULT 1`, `joined_at TEXT NOT NULL`, `updated_at`; `CHECK (role_in_group IN ('leader','co-leader','member','guest'))`, `CHECK (active IN (0,1))`. Unique index `group_memberships_active_uq (tenant_id, group_id, member_ref) WHERE active = 1`; index `group_memberships_tenant_group_idx (tenant_id, group_id)`.
- `attendance_records` — PK `(tenant_id, attendance_id)`; `occasion_ref TEXT NOT NULL`, `member_ref TEXT`, `status TEXT`, `headcount INTEGER`, `recorded_by_ref TEXT NOT NULL`, `recorded_at TEXT NOT NULL`, `updated_at`; `CHECK (status IS NULL OR status IN ('present','absent','excused'))`, `CHECK (headcount IS NULL OR headcount > 0)`, `CHECK ((member_ref IS NOT NULL AND status IS NOT NULL AND headcount IS NULL) OR (member_ref IS NULL AND headcount IS NOT NULL AND status IS NULL))`. Unique index `attendance_member_uq (tenant_id, occasion_ref, member_ref) WHERE member_ref IS NOT NULL`; index `attendance_tenant_occasion_idx (tenant_id, occasion_ref)`.
- `communication_messages` — PK `(tenant_id, message_id)`; `channel TEXT NOT NULL`, `subject TEXT`, `body_template TEXT NOT NULL`, `audience_json TEXT NOT NULL`, `status TEXT NOT NULL`, `origin TEXT NOT NULL`, `confirmed INTEGER NOT NULL DEFAULT 0`, `confirmed_by_ref TEXT`, `confirmation_reason TEXT`, `confirmed_at TEXT`, `created_by_ref TEXT NOT NULL`, `created_at`, `updated_at`; `CHECK (channel IN ('sms','email','push'))`, `CHECK (subject IS NULL OR channel = 'email')`, `CHECK (status IN ('draft','reviewed','confirmed','queued','sent','failed','canceled'))`, `CHECK (origin IN ('human','ai-drafted'))`, `CHECK (confirmed IN (0,1))`, `CHECK (status NOT IN ('confirmed','queued','sent') OR confirmed = 1)`, `CHECK (confirmed = 0 OR (confirmed_by_ref IS NOT NULL AND confirmation_reason IS NOT NULL AND confirmed_at IS NOT NULL))`, `CHECK (body_template <> '')`. Index `communication_messages_tenant_status_idx (tenant_id, status)`.
- `communication_recipients` — PK `(tenant_id, recipient_id)`; `message_id TEXT NOT NULL`, `member_ref TEXT NOT NULL`, `channel_ref TEXT NOT NULL`, `send_status TEXT NOT NULL`, `failure_reason TEXT`, `updated_at`; `CHECK (send_status IN ('pending','sent','delivered','failed','suppressed'))`, `CHECK (failure_reason IS NULL OR send_status = 'failed')`. Unique index `communication_recipients_uq (tenant_id, message_id, member_ref)`; index `communication_recipients_tenant_message_idx (tenant_id, message_id)`.
- `engagement_summaries` — PK `(tenant_id, summary_id)`; `scope_kind TEXT NOT NULL`, `member_ref TEXT`, `segment_ref TEXT`, `attendance_streak INTEGER NOT NULL DEFAULT 0`, `serving_count INTEGER NOT NULL DEFAULT 0`, `comms_response_count INTEGER NOT NULL DEFAULT 0`, `last_present_occasion_ref TEXT`, `window_start TEXT NOT NULL`, `window_end TEXT NOT NULL`, `computed_at TEXT NOT NULL`; `CHECK (scope_kind IN ('member','segment'))`, `CHECK ((scope_kind = 'member' AND member_ref IS NOT NULL AND segment_ref IS NULL) OR (scope_kind = 'segment' AND segment_ref IS NOT NULL AND member_ref IS NULL))`, `CHECK (attendance_streak >= 0)`, `CHECK (serving_count >= 0)`, `CHECK (comms_response_count >= 0)`, `CHECK (window_end >= window_start)`. Index `engagement_summaries_tenant_scope_idx (tenant_id, scope_kind)`.

## GraphQL surface
New file `apps/api/src/graphql/community.ts` (SDL + thin resolvers), merged into the executable schema, with `CommunityDomainError.code → extensions.code` mapping (mirrors `charts.ts` / `play.ts`). Resolvers parse input, authorize, and delegate — no logic. Enums: `MemberStatus`, `GroupKind`, `GroupRole`, `AttendanceStatus`, `CommunicationChannel`, `CommunicationStatus`, `CommunicationOrigin`, `RecipientSendStatus`, `ContactChannelKind`, `ConsentStatus`, `EngagementScopeKind`.

Types: `Member`, `Household`, `CommunityGroup`, `GroupMembership`, `AttendanceRecord`, `CommunicationMessage`, `CommunicationRecipient`, `EngagementSummary`, plus read-only projections `ResolvedAudience` and `AttendanceTally`. The GraphQL `Member` type exposes `contactChannelRefs` (refs + consent) but **never a raw contact value field** — the schema has no `phone`/`email`/`address` scalar on any Community+ type.

Queries:
`members(filter)` · `member(id)` · `households(filter)` · `household(id)` · `communityGroups(filter)` · `communityGroup(id)` · `groupMemberships(groupId)` · `attendanceRecords(occasionRef)` · `attendanceTally(occasionRef)` · `communicationMessages(filter)` · `communicationMessage(id)` · `communicationRecipients(messageId)` · `engagementSummaries(filter)` · `resolvedAudience(messageId)` *(preview the consent-filtered recipient set before confirming)*

Mutations:
- People/structure (non-outbound): `saveMember(input)` · `archiveMember(input)` *(soft-archive; explicit intent + audit)* · `saveHousehold(input)` · `saveCommunityGroup(input)` · `setGroupMembership(input)` · `removeGroupMembership(input)` *(destructive — `confirmationIntent { confirmed: true, reason }`, audited)* · `recordAttendance(input)` · `updateAttendance(input)`
- Communications (outbound lifecycle): `draftCommunicationMessage(input)` · `updateCommunicationMessage(input)` *(draft/reviewed only)* · `markCommunicationReviewed(input)` · **`confirmCommunicationSend(input)`** *(**human-confirmation gate** — requires `confirmationIntent { confirmed: true, reason }` + `confirmedByRef`; the only path that advances a message toward send; mirrors `removeChartAnnotation`'s confirmation intent but for **outbound comms to people**)* · **`queueConfirmedCommunication(input)`** *(hands the confirmed message + resolved consent-filtered audience to the send-integration adapter; **rejects unless the message is already `confirmed`**; never invoked by AI)* · `cancelCommunicationMessage(input)` *(explicit intent + audit)*
- Engagement: `recomputeEngagementSummaries(input)` *(idempotent recompute from attendance/membership/recipient rows; pure rollup behind it; writes only non-PII summaries)*

**Mutations that send communications to people are gated.** `confirmCommunicationSend` and `queueConfirmedCommunication` are the outbound-comms equivalents of stream start/stop + destructive mutations in the non-negotiables: they require explicit human `confirmationIntent`, record `confirmedByRef` + `reason` + `confirmedAt`, are audit-logged, and can never be driven by an AI suggestion path. `removeGroupMembership`, `archiveMember`, and `cancelCommunicationMessage` are destructive/irreversible-enough to also require explicit intent + audit.

WebSocket events emitted (durable, coarse; emitted only after durable state commits, each a `.strict()` payload added to the API event union in `apps/api/src/events/index.ts` with a tenant/aggregate-scope superRefine exactly like the presenter/play events): `community.memberUpdated` · `community.attendanceRecorded` · `community.communicationStatusChanged`. The comms event carries **status + ids only, never message body or recipient PII**.

## Service layer
Mirror the Charts/Play service split under `apps/api/src/services/community/`:
- `contracts.ts` / domain `contracts.ts` — `CommunityQueryService` + `CommunityCommandService` interfaces (Zod-validated request envelopes `{ actor, requestId, input }`), one method per query/mutation above.
- `errors.ts` — typed `CommunityDomainError` with `COMMUNITY_DOMAIN_ERROR_CODES` = `MEMBER_NOT_FOUND` · `HOUSEHOLD_NOT_FOUND` · `GROUP_NOT_FOUND` · `MEMBERSHIP_NOT_FOUND` · `ATTENDANCE_NOT_FOUND` · `MESSAGE_NOT_FOUND` · `RECIPIENT_NOT_FOUND` · `CONFIRMATION_REQUIRED` · `INVALID_LIFECYCLE_TRANSITION` · `CONSENT_REQUIRED` · `VALIDATION_FAILED` · `AUTHORIZATION_FAILED`, carrying `code` + redacted `safeMessage` (drives `extensions.code`).
- `in-memory.ts` — in-memory adapter implementing both services, tenant-scoped, Zod-validated, injected clock + id generator; the test double.
- `persistence.ts` — persistence-backed adapter delegating to the Community SQL repositories over an injected executor; a drop-in behind the same resolvers.
- `composition.ts` — `createCommunityPersistenceSelection` selecting `in-memory` vs `sql` by env (sql in production), applying `CommunityInitialSchemaMigration` via the shared migration runner; `migrateCommunitySqliteSchema` helper. Same `CommunityPersistenceSelectionConfig` shape as Charts/Play.

Every persisted read/write is tenant-scoped and role-checked; the command service owns its transaction; outbound-comms confirmation and destructive mutations require explicit intent + audit metadata; **the consent filter and lifecycle state machine are enforced in the service** (delegating to the pure functions) before any send-intent is produced; domain functions stay pure (the service only orchestrates, validates, clocks, audits, and persists). `queueConfirmedCommunication` calls a small injected `CommunicationSendPort` interface (an integration adapter, not implemented in this module) and stores the returned per-recipient status — the carrier SDK is isolated behind that port (API-shape rule).

## Offline posture (no sync queue in v1 — justified)
Community+ is **primarily online** and intentionally does **not** ship a local sync queue / replay coordinator in v1:
- The system-map failure strategy continues only **Play + Presenter** locally; the offline-first engineering rule names only **Play and Charts** as rehearsal-critical. Community+ admin work (managing members, recording attendance, sending comms) is not rehearsal-critical and is not in those lists.
- Outbound communications **must not** be queued-and-replayed blindly: replaying a send after a network gap risks **double-sending messages to people**, which violates the spirit of the human-confirm gate. Sends are deliberately online-only and idempotency-keyed at the integration boundary, not buffered offline.
- **Narrow exception considered, deferred:** field attendance capture (a volunteer marking who's present on a phone with poor signal) is the only plausibly offline-first Community+ flow. If the product wants it, it would reuse the **Charts/Play local-sync-queue + replay pattern** for the **non-destructive** `recordAttendance` / `updateAttendance` operations **only** (never comms), with `removeGroupMembership` / `archiveMember` / all comms excluded — exactly the destructive-exclusion posture Play uses for `removePlayCue`. Flagged as an open question, not built in v1.

## Privacy and safety (first-class)
This module handles **people and PII** and is the strictest privacy surface in the system. The non-negotiables apply with no exceptions:
- **PII segregation.** Raw contact values (phone, email, home address) are **never stored in Community+ domain records or tables** — only opaque `contactChannelRef`s resolved by a separate access-controlled contact-vault boundary. Giving/donation data and pastoral-care/counseling/prayer-request **note text** are **out of scope and never modeled here**. `[PII]`-marked fields (`displayName`, `Household.label`, custom-field values, contact refs) are segregated from every AI-bound projection.
- **No PII to AI.** Per `aiPolicyProfile`, no PII is sent to any third-party model unless `aiPolicyProfile.piiSharingAllowed = true`. By default Community+ AI features operate on the **`EngagementSummary` / segment-count / group-label** projection only — refs + counts + non-PII labels. The ChurchContext hard-exclusion list is honored absolutely: **never** to AI — phone numbers, home addresses, prayer/counseling notes, individual giving data, child-sensitive records.
- **Smallest projection.** AI callers request the smallest necessary `ChurchContext` projection — for Community+ that is the AI-safe `peopleProfile` (segment counts / anonymized labels) and `engagementProfile` (aggregates / trend labels) projections defined in the ChurchContext schema, plus `aiPolicyProfile`. Prompt functions accept only validated projections, never raw member/household/recipient records.
- **Human-confirmation gate for outbound comms.** Sending any communication to people requires explicit human confirmation (`confirmCommunicationSend` → `queueConfirmedCommunication`), recording `confirmedByRef` + `reason` + `confirmedAt`, audit-logged. AI may **draft** a message (`origin = "ai-drafted"`) but the draft cannot advance past `draft` without a human confirming — AI never sends.
- **Consent enforcement.** The pure audience resolver drops recipients whose `consentStatus ≠ granted` for the channel and flags them; the persisted recipient row records `suppressed`. No send is produced for a non-consented channel.
- **Tenant scope.** Required on every persisted member/household/group/membership/attendance/message/recipient/summary read and write; composite PKs lead with `tenant_id`; cross-tenant access throws `AUTHORIZATION_FAILED`.
- **Audit + no-secret logging.** Destructive mutations (`archiveMember`, `removeGroupMembership`, `cancelCommunicationMessage`) and every outbound-send confirmation are audited with actor + reason. Never log secrets, tokens, full message bodies, recipient contact values, or raw PII; `failureReason` / `safeMessage` surfaces are redacted only.
- **No credentials in records.** Carrier/vendor tokens (Twilio etc.) live in the integration adapter, never in Community+ domain records or tables.

## AI assist rules
- AI may suggest, as **reviewable** outputs only: a communication **draft** (subject/body template, `origin = "ai-drafted"`), an audience suggestion (by group/segment, expressed as refs), a re-engagement nudge list (members with lapsed attendance, **by ref**), or segment groupings. Output is Zod-validated and shown before any write, and **never** auto-advances a message toward send.
- AI operates on the **smallest necessary `ChurchContext` projection** — the AI-safe `peopleProfile`/`engagementProfile` aggregate projections + `aiPolicyProfile` — and on `EngagementSummary` rows (refs + counts). It receives **no** names, contact values, custom-field values, or note text unless `aiPolicyProfile.piiSharingAllowed = true`, and never the hard-excluded categories regardless.
- AI may rank/filter/segment existing members but may **not** invent members, households, or contact data, and must honor `aiPolicyProfile.enabledAIFeatures` + `humanReviewRequiredFor`.
- AI features return reviewable results before any write that affects people or comms (engineering rule: reviewable before writes that affect services, people, streams, or comms). Every prompt spec for this module declares purpose · required inputs · forbidden assumptions (no PII, no invented people) · output schema · fallback, and ships with a schema-compliance test + a fallback test.

## Open questions / risks / assumptions
- **ASSUMPTION — Community+ v1 feature shape.** The vision/system-map define Community+ only as "members · households · attendance · comms · engagement." This plan assumes v1 = directory + households + groups/memberships + attendance + an outbound-comms *draft→confirm→queue* lifecycle + non-PII engagement rollups, and **defers** the actual send transport, two-way inbox, giving, child check-in, and care/prayer-note content. Confirm this scope split.
- **ASSUMPTION — contact values live outside Community+ in an opaque contact-vault boundary** (members hold `contactChannelRef`s + consent only). This is the strongest privacy posture and mirrors Play media refs / Charts song refs, but the vault contract itself is **not designed here** and is a separate upcoming decision. If the product instead wants contact values stored in-module, that is a **material privacy decision** that must be made explicitly (and would still keep them out of every AI projection).
- **ASSUMPTION — comms send is an integration concern behind a `CommunicationSendPort`, gated by human confirmation, never queued-for-replay** (to avoid double-sends). Confirm the carrier(s) (Twilio SMS / email / push) and that online-only sending is acceptable.
- **ASSUMPTION — `EngagementSummary` is the AI-projectable record class** (refs + counts, PII-free by construction); all other Community+ records are PII-bearing and excluded from AI by default. Confirm the AI-safe projection boundary.
- **OPEN — offline attendance capture.** Should field attendance be offline-first (slice 13)? Default per the system map: **no** (Community+ is online-primary; only Play/Presenter/Charts are local-critical). Recommend deferring unless volunteers routinely record attendance with no signal.
- **OPEN — confirmation-gate granularity.** Is a per-message human confirmation sufficient, or do large audiences (e.g. > N recipients) warrant a second approval tier / role check? This plan gates every send once; confirm whether bulk sends need stronger controls.
- **OPEN — consent source of truth.** This plan stores `consentStatus` per channel ref on the member and enforces it in the audience resolver. Confirm whether consent is mastered in Community+ or in the contact-vault / an external compliance system (if external, Community+ reads it as an opaque flag).
- **RISK — PII leakage surface.** The single largest risk is PII reaching AI or logs. Mitigation is structural: PII-bearing records are typed `[PII]` and never enter a projection type that an AI caller can accept; the only AI-bound types (`EngagementSummary`, segment counts, group labels) cannot hold PII by construction; logs/queue surfaces carry redacted `safeMessage`s only. The plan recommends a dedicated test in slices 8–10 asserting the AI-bound projection types are PII-free.
- **RISK — double-send.** Mitigated by online-only sends, the confirmation gate, per-recipient idempotency keys at the integration boundary, and deliberately **excluding comms from any offline queue**.
- **DECISION — module sequence.** `NOW.md` sequences Community+ after Play and before OBS, backend-first. This plan assumes backend slices 1–10 land first; the web admin UI (slice 12) waits for the frontend-scaffold decision already recorded in `NOW.md`.
