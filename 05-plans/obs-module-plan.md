# OBS Module Plan

## Scope v1
Tenant-scoped **control surface for a church's own OBS Studio** (the production volunteer's livestream encoder), driven through **obs-websocket v5** behind an injected port · an **opaque connection profile** that names a reachable OBS instance by reference only (host/port/password live in a secret/vault boundary, never in the record) · a durable, coarse **scene / source / scene-item catalog snapshot** mirrored from OBS for display and validation · a durable, coarse **stream state** and **recording state** snapshot (active/inactive + last-known transition) · a first-class, **human-confirmation-gated `ObsActionIntent`** for every output-affecting action (start stream, stop stream, switch scene, toggle a source/scene-item) · an **action / audit log** of every requested and confirmed action with actor + reason · durable, coarse **OBS state events** (stream started/stopped, recording started/stopped, scene changed) on the API event union · graceful-degradation contracts so OBS being disconnected disables automation with visible status and never blocks Planning / Presenter / Play. Scene/source/stream **control is modeled as intent + confirmation in this domain**; the actual obs-websocket calls are realized by a desktop/agent runtime behind the port (mirrors how Play stores cue intent and the audio engine realizes it later).

## Out of scope v1
Generic livestream **hosting / CDN / RTMP destination management** (an explicit non-goal in the vision — Sanctuary OS controls the operator's encoder, it does not host or restream) · the **OBS process lifecycle** (launching, installing, updating, or supervising OBS itself; the operator runs OBS) · the **actual encoder / video pipeline / bitrate / GPU** (that is OBS's job, never modeled here) · **scene/source authoring** — creating, deleting, or re-laying-out scenes, sources, filters, transitions, or audio mixer levels inside OBS (v1 reads the catalog and switches/toggles existing scenes/sources; it does not design them) · **virtual camera, replay buffer, studio-mode preview/program**, and **profile/scene-collection management** (deferred; coarse stream/recording + scene switch only) · **high-frequency telemetry** (per-frame stats, dropped-frame counts, CPU/bitrate graphs, audio meters) — these stay on the local runtime, never on the durable event union or in SQLite · storing **OBS host / port / password / auth token / streaming-service stream key** in any domain record or table (held only as an opaque `connectionRef` resolved by a separate access-controlled secret/vault boundary; see Safety) · **multi-operator realtime co-control** of one OBS instance · **AI-initiated** stream or scene control (forbidden — AI may *suggest* a reviewable action intent, never confirm or fire one) · the **real obs-websocket client wiring** and the **operator UI** (desktop-runtime / UI work, deferred behind the port + the standing UI-surface decision).

## Domain objects
ObsConnectionProfile · Scene · Source · SceneItem · StreamState · RecordingState · ObsActionIntent · ObsActionLogEntry

## Domain boundaries
| Object | Owns | Does not own |
|---|---|---|
| ObsConnectionProfile | Tenant-scoped connection **identity**, a human label, an **opaque `connectionRef`** (vault handle), last-known `connectionStatus` (connected/disconnected/unknown), `obsWebsocketVersion?`, `lastSeenAt` | The OBS host / port / password / auth token / stream key (vault only), the OBS process, the network socket, the encoder |
| Scene | Tenant-scoped catalog snapshot of an OBS scene: `obsSceneRef` (OBS scene name/uuid as an opaque string), display name, `isCurrentProgramScene` (coarse last-known), ordering hint | The live OBS scene graph (OBS owns it), scene authoring, transitions, the rendered video |
| Source | Tenant-scoped catalog snapshot of an OBS source/input: `obsSourceRef`, kind label, `mutedHint?` (coarse), `activeHint?` (coarse) | Source filters, device handles, audio levels/mixer, raw media, the capture pipeline |
| SceneItem | Snapshot of a source's placement within a scene: `sceneRef` + `sourceRef` + `obsSceneItemId`, `visibleHint` (coarse last-known), ordering hint | Transform/crop/layout geometry, the live render, OBS scene-item authoring |
| StreamState | Durable, **coarse** stream snapshot per connection: `streamStatus` (active/inactive/unknown), `lastTransitionAt`, `lastTransitionActorId?`, `lastActionIntentRef?` | Bitrate, dropped frames, uptime ticks, per-frame stats (runtime-only), the RTMP destination / stream key |
| RecordingState | Durable, **coarse** recording snapshot per connection: `recordingStatus` (active/paused/inactive/unknown), `lastTransitionAt`, `lastTransitionActorId?` | The recording file path/bytes, disk, codec, the encoder |
| ObsActionIntent | A **requested output-affecting action**: `kind` (start-stream / stop-stream / switch-scene / toggle-source-visibility / toggle-source-mute), target refs, **`confirmation`** (`{ confirmed, confirmedByRef, reason, confirmedAt }`), lifecycle status (requested/confirmed/dispatched/succeeded/failed/canceled), `origin` (human / ai-suggested) | The obs-websocket transport (the port realizes it), OBS state itself, credentials |
| ObsActionLogEntry | Append-only audit row for one action attempt: action-intent ref, actor, reason, outcome, redacted `safeMessage?`, timestamps | The action decision logic, secrets, raw obs-websocket payloads/responses |

## API and storage surfaces
- GraphQL/HTTP is the cross-device control and persistence boundary for connection profiles, the scene/source/scene-item catalog snapshot, stream/recording state, action intents, and the action log (mirrors Charts / Play / Presenter / Community+).
- The **actual obs-websocket v5 session lives behind an injected `ObsControlPort`** (see *Injected port boundary*). The API service depends only on the port interface; the real client ships in **`packages/obs-agent`** and is driven by a **desktop/agent runtime** (analogous to the Presenter/Play desktop sidecar — ADR 0005). No real obs-websocket call ever happens in a unit test.
- **Per the system map, the OBS agent owns `obs-websocket, scene automation, human-confirm gate, logs`.** This module is the domain/persistence/API/event half of that ownership; the live socket + the agent runtime are the `packages/obs-agent` half.
- **No secrets in the persistence boundary.** A connection profile stores only an opaque `connectionRef` (a vault handle) + last-known status. The OBS host/port/password/auth token and any streaming-service stream key are resolved by a separate access-controlled secret/vault boundary (out of scope here, same posture as Community+ contact-vault refs / Play media refs / Charts song refs).
- Scene/source/scene-item rows are a **catalog snapshot mirrored from OBS** (refreshed via the port), never the source of truth — OBS is. Stream/recording rows are a **coarse, resumable** last-known snapshot (one row per connection), never a high-frequency feed.
- **Online-only for output-affecting actions.** Unlike Play/Charts, OBS is **not** offline-first: stream/scene control must reflect live OBS state and must never be queued-and-replayed (a replayed `start-stream` after a network gap could put a service live unattended — exactly the double-send hazard Community+ avoids for comms). Read-only catalog/state can be served from the last snapshot when OBS is disconnected, clearly flagged stale.

## Domain objects as records (Zod-shaped, tenant-scoped; no secrets; PII-free)
Every record is `.strict()`, branded-ID typed at boundaries, carries `tenantId`, and stores only **references and coarse state** — never the OBS host/port/password/token/stream key, never raw obs-websocket payloads, never PII (OBS controls production, not people; no PII is expected, and none is permitted). Field lists name the durable shape the API domain (`apps/api/src/domain/obs/contracts.ts`) and persistence records (`packages/db/src/obs-repository-contracts.ts`) must agree on.

- **ObsConnectionProfile** — `tenantId`, `connectionProfileId`, `label`, `connectionRef` (**opaque vault handle — never the host/port/password**), `connectionStatus` (enum: connected/disconnected/unknown), `obsWebsocketVersion?`, `lastSeenAt?`, `createdAt`, `updatedAt`.
  - Invariants: `label` non-empty; `connectionRef` is a reference, never a URL/credential; **no `host`/`port`/`password`/`token`/`streamKey` field exists on the schema at all** (structural, not just runtime).
- **Scene** — `tenantId`, `sceneId`, `connectionProfileId`, `obsSceneRef` (opaque OBS scene name/uuid), `displayName`, `isCurrentProgramScene` bool (coarse last-known), `orderHint` (nonneg int), `snapshotAt`.
  - Invariants: `displayName` non-empty; at most one `isCurrentProgramScene = true` per `(tenant, connectionProfileId)` (enforced in the service on snapshot apply); `obsSceneRef` opaque.
- **Source** — `tenantId`, `sourceId`, `connectionProfileId`, `obsSourceRef` (opaque), `kindLabel` (string, e.g. "browser_source" — descriptive only), `mutedHint?` bool, `activeHint?` bool, `snapshotAt`.
  - Invariants: hints are coarse last-known, nullable; no device handle or filter data; no audio level value.
- **SceneItem** — `tenantId`, `sceneItemId`, `connectionProfileId`, `sceneRef` (→ `obsSceneRef`), `sourceRef` (→ `obsSourceRef`), `obsSceneItemId` (opaque OBS numeric/string id), `visibleHint` bool (coarse), `orderHint` (nonneg int), `snapshotAt`.
  - Invariants: `(sceneRef, sourceRef, obsSceneItemId)` unique per `(tenant, connectionProfileId)`; `visibleHint` is last-known, not authoritative; no transform geometry.
- **StreamState** — `tenantId`, `connectionProfileId`, `streamStatus` (enum: active/inactive/unknown), `lastTransitionAt?`, `lastTransitionActorId?`, `lastActionIntentRef?`, `updatedAt`.
  - Invariants: one **coarse, resumable** row per `(tenant, connectionProfileId)`; **no bitrate / uptime / dropped-frame / per-frame fields** — high-frequency telemetry stays on the runtime bus, never here. A transition to `active`/`inactive` is only ever written by the service **after** a confirmed, dispatched `ObsActionIntent` succeeds (never speculatively).
- **RecordingState** — `tenantId`, `connectionProfileId`, `recordingStatus` (enum: active/paused/inactive/unknown), `lastTransitionAt?`, `lastTransitionActorId?`, `updatedAt`.
  - Invariants: one coarse row per `(tenant, connectionProfileId)`; no file path / bytes / codec; same "written only after a confirmed action succeeds (or a snapshot refresh)" rule.
- **ObsActionIntent** — `tenantId`, `actionIntentId`, `connectionProfileId`, `kind` (enum: `start-stream` / `stop-stream` / `switch-scene` / `toggle-source-visibility` / `toggle-source-mute`), `targetSceneRef?`, `targetSourceRef?`, `targetSceneItemId?`, `desiredVisible?` bool, `desiredMuted?` bool, `affectsLiveOutput` bool (**always `true` for every v1 kind** — see Safety), `status` (enum: requested/confirmed/dispatched/succeeded/failed/canceled), `origin` (enum: human / ai-suggested), `confirmation?: { confirmed: true, confirmedByRef, reason, confirmedAt }` (**required to leave `requested`**), `requestedByRef`, `createdAt`, `updatedAt`, `safeFailureMessage?` (redacted, present only when `status = failed`).
  - Invariants (superRefine): `kind = switch-scene` ⇒ `targetSceneRef` present; `kind = toggle-source-visibility` ⇒ `targetSourceRef` + `targetSceneItemId` + `desiredVisible` present; `kind = toggle-source-mute` ⇒ `targetSourceRef` + `desiredMuted` present; `start-stream`/`stop-stream` carry no target refs. **Status may advance to `confirmed`/`dispatched`/`succeeded` only if `confirmation.confirmed = true`.** `origin = "ai-suggested"` may exist only at `status = requested` and **can never self-advance** without a human `confirmation`. Allowed transition map enforced (requested→confirmed→dispatched→succeeded, with →canceled / →failed terminal branches); illegal jumps rejected. Holds **no socket handle, no obs-websocket payload, no credential**.
- **ObsActionLogEntry** — `tenantId`, `logEntryId`, `connectionProfileId`, `actionIntentRef`, `actorId`, `reason`, `outcome` (enum: requested/confirmed/dispatched/succeeded/failed/canceled), `safeMessage?` (redacted), `occurredAt`.
  - Invariants: append-only (no update path); `safeMessage` is redacted and present only for `failed`; never stores a raw obs-websocket response or any secret.

## Pure rendering/transform rules
The OBS analog of Charts `chordpro.ts` / Play `sequence.ts` / Community+ `message-lifecycle.ts` lives in `apps/api/src/domain/obs/` as pure, deterministic, Zod-validated, no-I/O functions (no `Date.now`, no randomness; clocks injected only at the service edge).

- **Action-eligibility / precondition checker** (`action-eligibility.ts`) is a pure function: `(ObsActionIntent, { connection: ObsConnectionProfile, scenes, sources, sceneItems, stream: StreamState, recording: RecordingState }) → ActionEligibility` — decides whether a requested action is *dispatchable* against the last-known snapshot **before** the port is touched, and flags (never throws) the blocking reason: connection not `connected`; `switch-scene` target `obsSceneRef` not in the scene snapshot; `toggle-*` target source/scene-item not in the snapshot; `start-stream` when `streamStatus = active` (already live — no-op/guard); `stop-stream` when `streamStatus = inactive`. Returns `{ eligible: boolean, reasons: ActionBlockReason[] }`, mirroring flagged-unresolved in Play and consent-exclusion flagging in Community+.
- **Action lifecycle transition** (`action-lifecycle.ts`) is a pure function: `(ObsActionIntent, transition, confirmation?) → ObsActionIntent | ActionLifecycleError` — validates the allowed-transition map **and** the *confirmation-required-to-dispatch* invariant as a discriminated-union state machine (per the engineering rule preferring discriminated unions for stateful workflows). It only **decides** whether the move is legal; it performs no dispatch and no I/O. This is the structural enforcement point of the human-confirm gate.
- **Scene-snapshot reconciliation** (`snapshot.ts`) is a pure function: `(previousCatalog, portObservedCatalog) → ReconciledCatalog` — diffs the last-known scene/source/scene-item snapshot against a freshly-observed one (from the port), producing add/update/remove deltas and a single `isCurrentProgramScene` resolution (at most one program scene). Deterministic; never mutates inputs; the service persists the result.
- **State-transition derivation** (`state.ts`) is a pure function mapping a confirmed-and-succeeded action (or an observed snapshot) to the next coarse `StreamState` / `RecordingState` — e.g. a succeeded `start-stream` ⇒ `streamStatus = active`, `lastTransitionAt = now(injected)`. Pure; the service supplies the clock and persists.
- Determinism + scope: all take already-tenant-scoped records and are tenant-agnostic at the function level; deterministic; injected clock only at the service edge.

## Injected port boundary (the obs-websocket seam)
The service depends on an **`ObsControlPort`** interface — **never** on `obs-websocket` directly. The real obs-websocket v5 client implements this port inside **`packages/obs-agent`** (and is driven by the desktop/agent runtime); a **fake** implements it in every unit test. This is the OBS analog of Community+'s injected `CommunicationSendPort` and follows the API-shape rule ("integration adapters isolate vendor SDKs and normalize failures").

- `ObsControlPort` (shape): `connect(connectionRef): Promise<ObsConnectionInfo>` · `disconnect(connectionRef): Promise<void>` · `getScenes(connectionRef): Promise<ObsObservedCatalog>` (scenes + sources + scene-items + current program scene) · `getStreamStatus(connectionRef): Promise<ObsObservedStreamStatus>` · `getRecordStatus(connectionRef): Promise<ObsObservedRecordStatus>` · `setCurrentProgramScene(connectionRef, obsSceneRef): Promise<void>` · `setSceneItemEnabled(connectionRef, { obsSceneRef, obsSceneItemId, enabled }): Promise<void>` · `setInputMute(connectionRef, { obsSourceRef, muted }): Promise<void>` · `startStream(connectionRef): Promise<void>` · `stopStream(connectionRef): Promise<void>` · `startRecord` / `stopRecord` / `pauseRecord` (optional v1).
- The port **takes a `connectionRef`**, not host/password — the **real implementation resolves the secret from the vault**; the credential never crosses the domain/API boundary and is never logged.
- The port **normalizes failures** to a typed, redacted shape (`{ retryable, safeMessage }`, aligned with `IntegrationFailure`); it never surfaces a raw obs-websocket error object, stack, or payload upward.
- **No real obs-websocket calls in unit tests.** Domain + service + resolver tests run entirely against an in-memory fake port (records calls, returns scripted catalogs/statuses, can simulate disconnect/failure). The real client is exercised only by an opt-in `packages/obs-agent` integration smoke against a developer's local OBS (mirrors the live-server / live-carrier gaps already flagged in `NOW.md`).

## Persistence model (SQLite-compatible: TEXT/INTEGER/REAL)
New file `packages/db/src/obs-migrations.ts` exporting `ObsInitialSchemaMigration` (via `defineSqlMigrationArtifact`), aggregated as `ObsSqlMigrations`, mirroring `charts-migrations.ts` / `play-migrations.ts` / `community-migrations.ts`. All tables tenant-scoped, composite PKs lead with `tenant_id`, `CHECK` constraints encode the Zod invariants, booleans as `INTEGER IN (0,1)`, enums/timestamps/opaque-refs as `TEXT`, order hints as `INTEGER`. **No column ever stores an OBS host, port, password, auth token, stream key, raw obs-websocket payload, or PII** — opaque refs and coarse state only. There is **no local-sync-queue migration** (OBS output actions are deliberately online-only — see *Offline posture*).

Initial schema tables (`schema_version` literal `obs.v1`):
- `obs_connection_profiles` — PK `(tenant_id, connection_profile_id)`; columns: `label TEXT NOT NULL`, `connection_ref TEXT NOT NULL`, `connection_status TEXT NOT NULL`, `obs_websocket_version TEXT`, `last_seen_at TEXT`, `schema_version TEXT NOT NULL`, `created_at TEXT NOT NULL`, `updated_at TEXT NOT NULL`; `CHECK (label <> '')`, `CHECK (connection_ref <> '')`, `CHECK (connection_status IN ('connected','disconnected','unknown'))`, `CHECK (schema_version = 'obs.v1')`. Index `obs_connection_profiles_tenant_idx (tenant_id)`. *(No host/port/password/token columns exist.)*
- `obs_scenes` — PK `(tenant_id, scene_id)`; `connection_profile_id TEXT NOT NULL`, `obs_scene_ref TEXT NOT NULL`, `display_name TEXT NOT NULL`, `is_current_program_scene INTEGER NOT NULL DEFAULT 0`, `order_hint INTEGER NOT NULL DEFAULT 0`, `snapshot_at TEXT NOT NULL`; `CHECK (display_name <> '')`, `CHECK (is_current_program_scene IN (0,1))`, `CHECK (order_hint >= 0)`. Unique index `obs_scenes_ref_uq (tenant_id, connection_profile_id, obs_scene_ref)`; index `obs_scenes_tenant_conn_idx (tenant_id, connection_profile_id)`.
- `obs_sources` — PK `(tenant_id, source_id)`; `connection_profile_id TEXT NOT NULL`, `obs_source_ref TEXT NOT NULL`, `kind_label TEXT NOT NULL`, `muted_hint INTEGER`, `active_hint INTEGER`, `snapshot_at TEXT NOT NULL`; `CHECK (muted_hint IS NULL OR muted_hint IN (0,1))`, `CHECK (active_hint IS NULL OR active_hint IN (0,1))`. Unique index `obs_sources_ref_uq (tenant_id, connection_profile_id, obs_source_ref)`; index `obs_sources_tenant_conn_idx (tenant_id, connection_profile_id)`.
- `obs_scene_items` — PK `(tenant_id, scene_item_id)`; `connection_profile_id TEXT NOT NULL`, `scene_ref TEXT NOT NULL`, `source_ref TEXT NOT NULL`, `obs_scene_item_id TEXT NOT NULL`, `visible_hint INTEGER NOT NULL DEFAULT 1`, `order_hint INTEGER NOT NULL DEFAULT 0`, `snapshot_at TEXT NOT NULL`; `CHECK (visible_hint IN (0,1))`, `CHECK (order_hint >= 0)`. Unique index `obs_scene_items_uq (tenant_id, connection_profile_id, scene_ref, source_ref, obs_scene_item_id)`; index `obs_scene_items_tenant_conn_scene_idx (tenant_id, connection_profile_id, scene_ref)`.
- `obs_stream_state` — PK `(tenant_id, connection_profile_id)` (one coarse, resumable row per connection); `stream_status TEXT NOT NULL`, `last_transition_at TEXT`, `last_transition_actor_id TEXT`, `last_action_intent_ref TEXT`, `updated_at TEXT NOT NULL`; `CHECK (stream_status IN ('active','inactive','unknown'))`. *(No bitrate/uptime/dropped-frame columns.)*
- `obs_recording_state` — PK `(tenant_id, connection_profile_id)`; `recording_status TEXT NOT NULL`, `last_transition_at TEXT`, `last_transition_actor_id TEXT`, `updated_at TEXT NOT NULL`; `CHECK (recording_status IN ('active','paused','inactive','unknown'))`.
- `obs_action_intents` — PK `(tenant_id, action_intent_id)`; `connection_profile_id TEXT NOT NULL`, `kind TEXT NOT NULL`, `target_scene_ref TEXT`, `target_source_ref TEXT`, `target_scene_item_id TEXT`, `desired_visible INTEGER`, `desired_muted INTEGER`, `affects_live_output INTEGER NOT NULL DEFAULT 1`, `status TEXT NOT NULL`, `origin TEXT NOT NULL`, `confirmed INTEGER NOT NULL DEFAULT 0`, `confirmed_by_ref TEXT`, `confirmation_reason TEXT`, `confirmed_at TEXT`, `requested_by_ref TEXT NOT NULL`, `safe_failure_message TEXT`, `created_at TEXT NOT NULL`, `updated_at TEXT NOT NULL`;
  `CHECK (kind IN ('start-stream','stop-stream','switch-scene','toggle-source-visibility','toggle-source-mute'))`,
  `CHECK (status IN ('requested','confirmed','dispatched','succeeded','failed','canceled'))`,
  `CHECK (origin IN ('human','ai-suggested'))`,
  `CHECK (affects_live_output IN (0,1))`, `CHECK (confirmed IN (0,1))`,
  `CHECK (desired_visible IS NULL OR desired_visible IN (0,1))`, `CHECK (desired_muted IS NULL OR desired_muted IN (0,1))`,
  `CHECK (kind <> 'switch-scene' OR target_scene_ref IS NOT NULL)`,
  `CHECK (kind <> 'toggle-source-visibility' OR (target_source_ref IS NOT NULL AND target_scene_item_id IS NOT NULL AND desired_visible IS NOT NULL))`,
  `CHECK (kind <> 'toggle-source-mute' OR (target_source_ref IS NOT NULL AND desired_muted IS NOT NULL))`,
  `CHECK (status NOT IN ('confirmed','dispatched','succeeded') OR confirmed = 1)`,
  `CHECK (confirmed = 0 OR (confirmed_by_ref IS NOT NULL AND confirmation_reason IS NOT NULL AND confirmed_at IS NOT NULL))`,
  `CHECK (safe_failure_message IS NULL OR status = 'failed')`.
  Index `obs_action_intents_tenant_conn_status_idx (tenant_id, connection_profile_id, status)`.
- `obs_action_log` — PK `(tenant_id, log_entry_id)`; `connection_profile_id TEXT NOT NULL`, `action_intent_ref TEXT NOT NULL`, `actor_id TEXT NOT NULL`, `reason TEXT NOT NULL`, `outcome TEXT NOT NULL`, `safe_message TEXT`, `occurred_at TEXT NOT NULL`; `CHECK (outcome IN ('requested','confirmed','dispatched','succeeded','failed','canceled'))`, `CHECK (safe_message IS NULL OR outcome = 'failed')`. Index `obs_action_log_tenant_conn_idx (tenant_id, connection_profile_id)`; index `obs_action_log_intent_idx (tenant_id, action_intent_ref)`.

Persistence contracts (`packages/db/src/obs-repository-contracts.ts`, mirroring `charts-repository-contracts.ts` / `community-repository-contracts.ts`):
- `ObsPersistenceReadOptions` / `ObsPersistenceWriteOptions` reuse `RepositoryReadOptions` / `RepositoryWriteOptions` and require an `actorId` (superRefine) so every read/write is attributable for audit.
- One `*PersistenceRecordSchema` per object + `*PersistenceInputSchema` per operation; `readOperation` / `writeOperation` wrappers; `ObsQueryPersistenceRepository` + `ObsCommandPersistenceRepository` interfaces. The connection-profile persistence record schema **re-asserts the no-secret posture** (it has no host/port/password/token field; `connectionRef` is opaque).

Migration artifact: `ObsInitialSchemaMigration` via `defineSqlMigrationArtifact` (eight tables + indexes, all `CHECK`s, `tenantScopedTables` = all eight, `transactional: true`), aggregated as `ObsSqlMigrations`. No local-sync-queue migration in v1 (see *Offline posture*).

## GraphQL surface
New file `apps/api/src/graphql/obs.ts` (SDL + thin resolvers), merged into the executable schema, with `ObsDomainError.code → extensions.code` mapping (mirrors `charts.ts` / `play.ts` / `community.ts`). Resolvers parse input, authorize, and delegate — no logic. Enums: `ObsConnectionStatus`, `ObsStreamStatus`, `ObsRecordingStatus`, `ObsActionKind`, `ObsActionStatus`, `ObsActionOrigin`, `ObsActionLogOutcome`.

Types: `ObsConnectionProfile`, `ObsScene`, `ObsSource`, `ObsSceneItem`, `ObsStreamState`, `ObsRecordingState`, `ObsActionIntent`, `ObsActionLogEntry`, plus a read-only `ObsActionEligibility` projection (the pure precondition result, previewable before confirming). **No type exposes a host/port/password/token/stream-key field** — the schema has no such scalar anywhere on the OBS surface.

Queries (read-only — safe whether OBS is connected or showing a stale snapshot):
`obsConnectionProfiles(filter)` · `obsConnectionProfile(id)` · `obsScenes(connectionProfileId)` · `obsSources(connectionProfileId)` · `obsSceneItems(connectionProfileId, sceneRef?)` · `obsStreamState(connectionProfileId)` · `obsRecordingState(connectionProfileId)` · `obsActionIntents(filter)` · `obsActionLog(connectionProfileId)` · `obsActionEligibility(actionIntentInput)` *(preview the pure precondition check before requesting/confirming)*

Mutations:
- Connection / catalog (non-output-affecting, still tenant-scoped + audited): `saveObsConnectionProfile(input)` *(stores `label` + opaque `connectionRef` only — rejects any attempt to pass a host/password)* · `removeObsConnectionProfile(input)` *(destructive — `confirmationIntent { confirmed: true, reason }`, audited)* · `refreshObsCatalog(input)` *(asks the port for the live catalog/status and reconciles the snapshot; online-only; reads OBS, changes no OBS state)*
- **Output-affecting actions — every one is human-confirm-gated, online-only, audited, never AI-driven:**
  - `requestObsAction(input)` *(creates an `ObsActionIntent` at `status = requested`; runs the pure eligibility check; **does not touch OBS** — nothing is dispatched yet; `origin = human` or, for an AI suggestion, `ai-suggested`)*
  - **`confirmObsAction(input)`** *(**the human-confirmation gate** — requires `confirmationIntent { confirmed: true, reason }` + `confirmedByRef`; advances `requested → confirmed`; the only path that authorizes a dispatch; the OBS analog of Community+ `confirmCommunicationSend` and the Charts/Play destructive-confirmation intents)*
  - **`dispatchObsAction(input)`** *(executes the confirmed intent through the injected `ObsControlPort` — `startStream` / `stopStream` / `setCurrentProgramScene` / `setSceneItemEnabled` / `setInputMute`; **rejects unless `status = confirmed`**; writes the resulting coarse `StreamState`/`RecordingState`/scene snapshot and the action-log outcome; never invoked by AI)*
  - `cancelObsAction(input)` *(explicit intent + audit; terminal `requested`/`confirmed` → `canceled`)*

  > `requestObsAction` + `confirmObsAction` + `dispatchObsAction` is a deliberate **three-step gate**: request (proposes + precondition-checks, no OBS contact), confirm (human authorizes with a reason — the gate), dispatch (the single moment OBS state changes). `confirmObsAction`/`dispatchObsAction` can never be reached by an AI path; AI may only create a `requested`, `origin = "ai-suggested"` intent for a human to confirm. `removeObsConnectionProfile` is destructive and also requires explicit intent + audit.

WebSocket events emitted (durable, coarse — high-frequency telemetry stays on the local runtime bus): `obs.connectionStatusChanged` · `obs.streamStateChanged` · `obs.recordingStateChanged` · `obs.sceneChanged` · `obs.actionStatusChanged`. Each is a `.strict()` payload added to the API event union (`apps/api/src/events/index.ts`) with a tenant/aggregate-scope superRefine exactly like the presenter/play/community events, **carrying status + ids + actor only — never a credential, raw payload, bitrate, or any high-frequency stat** — and emitted only after durable state commits. Per-frame/bitrate telemetry is **intentionally excluded** from the union.

## Service layer
Mirror the Charts / Play / Community+ service split under `apps/api/src/services/obs/`:
- `contracts.ts` / domain `contracts.ts` — `ObsQueryService` + `ObsCommandService` interfaces (Zod-validated request envelopes `{ actor, requestId, input }`), one method per query/mutation above.
- `errors.ts` — typed `ObsDomainError` with `OBS_DOMAIN_ERROR_CODES` = `CONNECTION_PROFILE_NOT_FOUND` · `SCENE_NOT_FOUND` · `SOURCE_NOT_FOUND` · `SCENE_ITEM_NOT_FOUND` · `STREAM_STATE_NOT_FOUND` · `RECORDING_STATE_NOT_FOUND` · `ACTION_INTENT_NOT_FOUND` · `CONFIRMATION_REQUIRED` · `INVALID_ACTION_TRANSITION` · `ACTION_NOT_ELIGIBLE` · `OBS_DISCONNECTED` · `PORT_FAILURE` · `VALIDATION_FAILED` · `AUTHORIZATION_FAILED`, carrying `code` + redacted `safeMessage` (drives `extensions.code`; `PORT_FAILURE` wraps the normalized port failure without leaking the raw error).
- `in-memory.ts` — in-memory adapter implementing both services, tenant-scoped, Zod-validated, injected clock + id generator, **injected `ObsControlPort`** (defaults to a no-op/disconnected fake); the test double.
- `persistence.ts` — persistence-backed adapter delegating to the OBS SQL repositories over an injected executor + the same injected `ObsControlPort`; a drop-in behind the same resolvers.
- `composition.ts` — `createObsPersistenceSelection` selecting `in-memory` vs `sql` by env (sql in production), applying `ObsInitialSchemaMigration` via the shared migration runner; `migrateObsSqliteSchema` helper. Same `ObsPersistenceSelectionConfig` shape as Charts / Play / Community+, plus an injected `ObsControlPort` slot (real impl supplied by the desktop/agent runtime; fake in tests).

Every persisted read/write is tenant-scoped and role-checked; the command service owns its transaction; **the confirmation gate and the action state machine are enforced in the service** (delegating to the pure `action-lifecycle.ts` + `action-eligibility.ts`) **before any `ObsControlPort` call** — `dispatchObsAction` refuses to call the port unless the intent is `confirmed`; destructive `removeObsConnectionProfile` requires explicit intent + audit; domain functions stay pure (the service only orchestrates, validates, clocks, audits, persists, and — for dispatch/refresh only — calls the port). Every output-affecting action writes an `ObsActionLogEntry` at each lifecycle step.

## Offline posture (no sync queue in v1 — justified)
OBS is **online-only for output-affecting actions** and intentionally does **not** ship a local sync queue / replay coordinator:
- The system-map failure strategy continues only **Play + Presenter** locally; the offline-first engineering rule names only **Play and Charts** as rehearsal-critical. OBS control is neither, and "OBS disconnected ⇒ automation disabled with visible status" is the *prescribed* behavior — degrade, don't buffer.
- Output actions **must not** be queued-and-replayed: replaying a `start-stream`/`stop-stream`/`switch-scene` after a network or socket gap could take a service **live (or dark) unattended**, violating the spirit of the human-confirm gate — exactly the double-send hazard Community+ avoids for comms. Output actions are deliberately online-only and require a fresh confirmation.
- **Read-only catalog/state stays available stale.** When OBS is disconnected, the last-known scene/source/scene-item and coarse stream/recording snapshots remain queryable, **clearly flagged stale** (`connectionStatus = disconnected`, `snapshotAt` shown), so Planning/Presenter/Play are never blocked (system-map contract). No write to OBS is attempted while disconnected; eligibility returns `ineligible` with an `OBS_DISCONNECTED` reason.

## Safety (first-class)
This module controls **live, public-facing output** and is the system's strongest "automation must fail gracefully" surface. The non-negotiables apply with no exceptions:
- **Human-confirmation gate on every output-affecting action.** Stream start, stream stop, scene switch, source visibility toggle, and source mute toggle **all** carry `affectsLiveOutput = true` and **all** require an explicit human `confirmationIntent { confirmed: true, reason }` + `confirmedByRef`, recorded with `confirmedAt` and audit-logged, before the service will dispatch through the port. The three-step `request → confirm → dispatch` flow makes the gate structural: no single mutation both decides and goes live. This is the OBS realization of the engineering-rule non-negotiable and matches the Community+ comms-send gate + the Charts/Play destructive-confirmation intents.
- **Never auto-start/stop a stream or auto-switch a scene.** There is **no** code path — scheduled, event-driven, or AI — that confirms or dispatches an OBS action without a human. AI may create only a `requested`, `origin = "ai-suggested"` intent; it can never reach `confirmObsAction`/`dispatchObsAction`. (Per the vision: "AI assists, not decides"; "every automation path must fail gracefully.")
- **No secrets/credentials in domain records.** OBS host/port/password/auth token and any streaming-service stream key live **only** in a separate access-controlled secret/vault boundary, referenced by an opaque `connectionRef`. **No domain schema or DB column holds them** (structural, verified by a test asserting the absence of such fields). The port resolves the secret from the vault at call time; it is never logged, never returned upward, never placed on an event.
- **Tenant scope.** Required on every persisted connection/scene/source/scene-item/state/intent/log read and write; composite PKs lead with `tenant_id`; cross-tenant access throws `AUTHORIZATION_FAILED`.
- **Audit + no-secret logging.** Every requested/confirmed/dispatched/succeeded/failed/canceled action and the destructive `removeObsConnectionProfile` are audited with actor + reason in `obs_action_log`. Never log secrets, tokens, raw obs-websocket payloads, or stream keys; `safeMessage` / `safeFailureMessage` surfaces are redacted only (the port normalizes failures to a safe shape).
- **Graceful degradation.** OBS disconnection disables actions with **visible status** (`connectionStatus`, eligibility reasons) and never blocks Planning / Presenter / Play (system-map contract). Read snapshots remain available, flagged stale.
- **No PII.** OBS controls production hardware/scenes, not people; no PII is expected in any field, and the schemas/types provide nowhere to put it.

## AI assist rules
- AI may, as a **reviewable suggestion only**, propose a *next* action — e.g. "switch to the `Lower Third` scene for the announcements" — expressed strictly as a `requested`, `origin = "ai-suggested"` `ObsActionIntent` (refs only). It is Zod-validated and shown for human confirmation; it **can never** advance itself toward dispatch, and AI **never** starts/stops a stream or switches a scene directly (engineering rule: reviewable before writes that affect *streams*; vision: AI assists, not decides).
- AI callers request the **smallest necessary `ChurchContext` projection** (e.g. service order / segment labels to reason about *when* a scene change might help). No credentials, no stream keys, and no raw obs-websocket state are ever placed in a prompt; OBS records carry no PII regardless.
- AI must honor `aiPolicyProfile.enabledAIFeatures` + `humanReviewRequiredFor`; every prompt spec for this module declares purpose · required inputs · forbidden assumptions (never confirm/dispatch, never emit credentials, never invent scenes/sources not in the snapshot) · output schema (a `requested` action intent) · fallback, and ships with a schema-compliance test + a fallback test.

## Open questions / risks / assumptions
- **ASSUMPTION — OBS v1 = control of the operator's *own* OBS, not hosting.** The vision lists "livestream" as a connected workflow but names "generic livestream hosting (v1)" a non-goal. This plan assumes v1 controls a church's existing OBS Studio via obs-websocket v5 (scene switch, source toggle, coarse stream/record start/stop) and **defers** any CDN/RTMP destination, restreaming, scene authoring, studio-mode, and virtual-camera. Confirm this scope.
- **ASSUMPTION — every v1 output action is human-confirm-gated (`affectsLiveOutput = true` for all of `start-stream`/`stop-stream`/`switch-scene`/`toggle-source-visibility`/`toggle-source-mute`).** The engineering rule names "stream start/stop and OBS scene/source automation"; this plan reads "automation" broadly and gates **every** action that changes what viewers see/hear, including scene switches and source toggles — not only stream start/stop. Confirm whether any low-risk action (e.g. muting a not-yet-live source while `streamStatus = inactive`) may be exempted, or whether the uniform gate is preferred (this plan recommends the uniform gate for safety and simplicity).
- **ASSUMPTION — OBS credentials live in a separate secret/vault boundary, referenced by an opaque `connectionRef`** (records/columns hold no host/port/password/token/stream key). This is the strongest posture and mirrors Community+ contact-vault / Play media / Charts song refs, but the **vault contract itself is not designed here** and is a separate upcoming decision (where do OBS connection secrets live, and who can read them).
- **ASSUMPTION — stream/recording state is a coarse, resumable snapshot; high-frequency telemetry (bitrate, dropped frames, uptime, audio meters) stays on the local runtime bus**, never on the durable event union or in SQLite (mirrors Play's `PlaybackState` vs per-sample playhead split, and the system-map "local desktop event bus — high-frequency sync"). Confirm the durable granularity (status + last transition) is the right persistence boundary.
- **ASSUMPTION — control is stored as *intent + confirmation*; the obs-websocket calls are realized by the agent runtime behind the injected port** (mirrors Play storing cue intent for the audio engine to realize). The real client + runtime are slices 11–12, gated on an OBS connection decision.
- **OPEN — three-step (`request → confirm → dispatch`) vs two-step gate.** This plan uses three steps so confirmation is structurally separate from going live (no mutation both decides and dispatches). A two-step (`requestAndConfirm` then `dispatch`, or a single confirmed-dispatch) is simpler but couples the gate to the side effect. Confirm the three-step gate is acceptable, or collapse to two with the confirmation still mandatory.
- **OPEN — confirmation granularity / re-confirmation.** Does each dispatch need its own fresh confirmation, or may a confirmed intent be re-dispatched (e.g. retry after a transient port failure) without re-confirming? This plan requires `status = confirmed` to dispatch and treats a failed dispatch as terminal `failed` (a new intent + new confirmation is needed to retry), avoiding an unattended auto-retry going live. Confirm the retry posture.
- **OPEN — `obs-agent` package shape + obs-websocket version pin.** `packages/obs-agent` is currently a stub ("Prepared for future work") and the `obs-integrator` skill is a stub ("Expand when role is active"); this plan defines the agent's intended role (own the live obs-websocket v5 session behind the port, refresh snapshots, dispatch confirmed intents) but the concrete package layout, the obs-websocket v5 version pin, and the Tauri sidecar supervision await the desktop-shell decision.
- **RISK — an action goes live unattended.** The single largest risk. Mitigated structurally: the three-step gate, `confirmation.confirmed = true` required to leave `requested`, the service refusing to call the port unless `confirmed`, online-only with **no** offline replay of output actions, failed-dispatch-is-terminal (no auto-retry), and AI confined to `requested`-only suggestions. The plan recommends a dedicated test in slices 7–9 asserting **no reachable path dispatches without a human confirmation**.
- **RISK — secret leakage.** Mitigated structurally: no secret field/column exists (verified by tests), the port resolves the vault handle and never returns/logs it, failures are normalized to redacted `safeMessage`s, and events/logs carry ids + status only.
- **DECISION — module sequence.** `NOW.md` sequences OBS as the **final** module, backend-first; after OBS the remaining build is the deferred UIs + live integration/runtime slices (OBS connection, carrier send, desktop shell) — all of which need the user (surface, visual verification, external-account/connection decisions). This plan assumes backend slices 1–10 land first; slices 11–13 wait for those user decisions.
